01 · The lifecycle

Four events fire in order.
Your agent never sees the seams.

From the moment you press Enter to the moment Claude writes back, engram is watching four Claude Code lifecycle events. Three of them inject context; one just takes notes. Read left to right.

fires first 01

SessionStart

A project brief lands before you type a word.

When Claude Code opens in an engram-initialized project, the Sentinel reads .git/HEAD and queries the graph for god nodes, stats, and top landmines. It composes a ~500-token project brief and hands it to Claude as additionalContext.

Source guard: fires on startup · clear · compact. Resume is passthrough so context isn’t double-injected.

replaces ~4 exploration reads · ≈ −5,000 tok
fires on every prompt 02

UserPromptSubmit

Your prompt is pre-queried against the graph.

Keywords are extracted (≥3 chars, non-stopword). If there are ≥2 significant terms AND the graph returns ≥3 matching nodes, engram pre-loads the relevant subgraph as additionalContext. Claude starts reasoning with structure already in hand.

Privacy invariant: raw prompt text is never written to the hook log. Only the decision (injected / skipped) is recorded. Asserted in tests.

prevents ~1.5 reads per prompt · ≈ −8,000 tok
fires on every tool call 03

PreToolUse

Read, Edit, Write, and Bash are intercepted.

Read: deny + reason with structural summary. File is never actually read — the summary arrives as a system-reminder containing the graph view. Edit/Write: allow + landmine warnings (never blocks). Bash: strict parser for cat/head/tail single-file invocations delegates to the Read handler.

The money path. 60% hit rate on mature projects. This is where 18,000 of the 42,500 tokens come from.

replaces full reads · ≈ −18,000 tok
fires on every completion 04

PostToolUse

Pure observer — logs, never injects.

After the tool runs, the Sentinel records tool name, file path, output size, success flag, and the PreToolUse decision to .engram/hook-log.jsonl. The log is append-only with 10 MB rotation and atomic writes on POSIX.

Why observer only? v0.3.0 produces the data; v0.3.1 will use it to self-tune the confidence threshold, mistake boost, and coverage ceiling. Never modifies tool output.

no direct savings · enables self-tuning

Every Read goes through nine checks before engram decides.

The Read handler is the highest-leverage component. Nine passthrough branches guard against false-positive interceptions. The tenth branch is the one that saves tokens.

03 · The substrate

Six parallel layers feed the same session.

engram is one of six persistence layers your Claude Code session leans on. Each one fills a different gap. The Sentinel is the newest; the others have been running for months.

layer 01

engram graph

.engram/graph.db · SQLite

Structural code knowledge. AST miner (10 languages, zero LLM cost), git miner (co-change, hot files), session miner (decisions, mistakes). v0.3 sentinel

layer 02

MemPalace

~/.mempalace · ChromaDB + SQLite KG

Semantic memory across 8,646 drawers. Mined from vault, projects, rules, agent memory. Hourly miner + real-time fswatch. always on

layer 03

Rule files

~/.claude/rules/ · 9 files

brain-os, coding-style, context-rules, failure-playbook, hooks, super-agents, agent-memory, mcp-cli, unified-memory. Loaded every session.

layer 04

Skills

~/.claude/skills/ · 418 skills

Pattern-matched, lazy-loaded. Indexed into engram by the skills-miner as concept nodes with triggered_by edges to keyword concepts.

layer 05

Agents

~/.claude/agents/ · 101 agents

Custom + plugin + super commanders. Each spawn reads its own memory file in ~/vault/01-KNOWLEDGE/agent-memory/ before work begins.

layer 06

MCP servers

mcp-cli wrappers (T1) · direct MCP (T4)

context7, mempalace, obsidian, render, sentry, playwright, engram, contextplus. CLI wrappers preferred — ~100 tok vs ~5K tok direct call.

05 · Quick reference

Seven commands you’ll actually use.

Everything else is a passthrough branch or a safety invariant. These seven are the surface you interact with.

install
engram install-hook

Adds the four hook entries to .claude/settings.local.json. Atomic. Idempotent. Backed up.

dry-run
engram hook-preview <file>

Show what the Read handler would do for a file. Deny, allow, or passthrough — with the reason.

measure
engram hook-stats

Summarize .engram/hook-log.jsonl. Per-event, per-tool, per-decision breakdown. Estimated tokens saved.

kill switch
engram hook-disable

Touch .engram/hook-disabled. Every handler checks it and passes through. Reversible with hook-enable.

rebuild graph
engram init

Re-mine the project. ~40 ms, zero LLM calls, zero tokens used. Run after structural changes.

explicit query
engram query "..."

Direct graph query. BFS with 2,000-token budget. For when the hook didn’t catch what you needed.

rollback
engram uninstall-hook

Surgical removal. Preserves every other hook in settings.json. Automatic backup before the write.

upgrade
npm i -g engramx@latest

Pull the latest from npm. v0.3.1 will tune confidence thresholds from real hook-log data.
