Integration Guide

Make your tool better
with engram underneath

engram is a structural code graph that intercepts AI agent tool calls. When your memory tool, compression plugin, or workflow system sits on top of engram, both get better. Here's how — with real numbers.

The Problem Every Tool Shares

Every AI coding tool burns tokens re-reading files the agent already understands. Memory tools, context compressors, and workflow managers all sit after the read — they compress or index what was already loaded. The tokens are already spent.

Without engram

With engram

engram intercepts Read, Edit, Write, and Bash calls before they execute. When the graph has confident coverage of a file, the raw read never happens — the agent sees a ~300-token structural summary instead of ~1,500 tokens of raw code.

Where engram Sits in the Stack

Your AI Agent Claude Code, Cursor, Windsurf, Copilot

engram (Sentinel hooks) structural interception layer

engram sits between the agent and the filesystem. Your tool sits above engram. When both are present, your tool processes already-compressed structural context instead of raw file contents. Less noise in, better output out.

Integration Patterns

Three ways your tool can integrate with engram, from easiest to deepest:

Ready now

Pattern 1: Query the graph via CLI

Shell out to engram and parse the output. Zero dependencies. Works from any language.

# From your tool's code (any language that can shell out)
engram query "authentication flow" -p /path/to/project
engram gods -p /path/to/project
engram stats -p /path/to/project
engram mistakes -p /path/to/project
engram path "auth" "database" -p /path/to/project

Use case: Your memory tool wants to know the project's core entities before storing a decision. Query engram gods and tag the memory with the relevant module names.

Ready now

Pattern 2: Import the programmatic API

If your tool is Node.js/TypeScript, import engram directly. No subprocess overhead.

import {
  query, godNodes, stats, mistakes,
  getFileContext, getStore
} from "engramx";

// Get structural context for a file
const ctx = await getFileContext("/path/to/project", "src/auth.ts");
// ctx.summary → ~300-token structural summary
// ctx.nodeCount → number of nodes for this file
// ctx.confidence → graph coverage confidence (0-1)

// Query the graph
const result = await query("/path/to/project", "how does auth work");
// result.text → budget-capped structural summary
// result.nodesFound → number of matching nodes

// Get the raw graph store
const store = await getStore("/path/to/project");
const allNodes = store.getAllNodes();
// Full access to nodes, edges, metadata
store.close();

Use case: Your compression plugin checks engram's confidence before compressing a file. If confidence ≥ 0.7, skip compression entirely — engram already handled it at the hook layer.

Planned v0.5

Pattern 3: Hook chain (sub-hooks)

Register your tool as a sub-hook inside engram's dispatch. engram intercepts first, then forwards to your handler with the structural context already attached.

// Future: engram.config.json
{
  "subHooks": {
    "PreToolUse:Read": [
      { "command": "your-tool intercept", "timeout": 3 }
    ]
  }
}

Use case: Your code review tool wants to see what files were read during a session. Instead of installing a separate PostToolUse hook (risking conflicts), register as an engram sub-hook and get the read events with structural context pre-attached.

Ready now

Pattern 4: Bundle into SessionStart

engram's SessionStart handler already queries external tools and bundles their output. If your tool has a CLI, engram can query it and include the results in the session brief.

// engram v0.4 already does this with mempalace:
// 1. SessionStart fires
// 2. engram queries graph (god nodes, stats, landmines)
// 3. engram queries mcp-mempalace (semantic findings)
// 4. Both results bundled into one additionalContext injection
//
// Your tool can follow the same pattern.
// The agent sees one unified brief instead of N separate injections.

Use case: Your cross-session memory tool has findings about the current project. Instead of a separate SessionStart hook that might conflict with engram's, contribute your context to engram's brief. One injection, both layers.

Integration Guides by Tool Type

Memory tools (mem0, mempalace, Claude-Mem, ClawMem)

What you store: decisions, learnings, session context, conversation history.
What engram stores: code structure, call graphs, git history, past mistakes.
The combination: Your tool says "we chose JWT for auth last week." engram says "AuthService lives in src/auth.ts, calls TokenValidator, has 3 known landmines." Together, the agent has both the why and the where.

Integration effort

~50 lines. Query engram gods -p $PROJECT at session start, tag your memories with the module names. Or import getFileContext and check structural coverage before storing file-level context (skip files engram already covers).

Context compression tools (context-mode, toonify, token-reducer)

What you do: compress tool output after the read.
What engram does: prevent the read from happening.
The combination: engram intercepts 60-75% of reads. Your tool compresses the remaining 25-40% that pass through. Sequential, not competing.

Code review tools (CodeGraph, code-review-graph, Graphify)

What you build: code knowledge graphs for review context.
What engram adds: mistake memory (past bugs on the same files), git-mined hot files, and session-learned patterns.
The combination: Your review tool checks structure. engram warns "this file had a race condition fixed 3 days ago — verify the fix still holds."

Workflow tools (Continuous-Claude, Gobby, eywa-continuum)

What you manage: session continuity, state handoffs, multi-agent coordination.
What engram adds: PreCompact hook ensures structural context survives compaction. CwdChanged hook auto-switches project context. Your handoff payloads can include engram's god nodes instead of raw file lists.

Hook Coexistence

engram registers 6 hook events in Claude Code. Here's exactly what it uses and how your hooks can coexist:

Hook Eventengram's BehaviorYour Tool
PreToolUse Intercepts Read/Edit/Write/Bash only (matcher: Read|Edit|Write|Bash) Use a different matcher (e.g., Glob|Grep) — no conflict
PostToolUse Observer only (logs decisions, never modifies) Safe to coexist — multiple PostToolUse hooks run independently
SessionStart Injects project brief as additionalContext Your SessionStart hook runs separately — both contexts get injected
UserPromptSubmit Pre-queries graph based on message keywords (TF-IDF filtered) Safe to coexist — multiple injections stack
PreCompact Re-injects critical structural context before compaction Your PreCompact hook can inject its own survival payload alongside
CwdChanged Switches project graph on directory change Your hook can read engram's new project root from the payload
No conflicts

engram's PreToolUse matcher is Read|Edit|Write|Bash. If your tool hooks Glob, Grep, WebSearch, or any other tool, there's zero overlap. Claude Code runs all matching hooks in order.

Quick Start for Integration

# 1. Install engram in your test project
npm install -g engramx
cd ~/your-test-project
engram init
engram install-hook

# 2. Query the graph from your tool
engram query "main entry point" -p .
engram gods -p .
engram stats -p .

# 3. Check if a file is covered (for conditional compression)
engram hook-preview src/auth.ts

# 4. Start the file watcher (live re-indexing)
engram watch

The graph is a local SQLite file at .engram/graph.db. Zero cloud. Zero API keys. Zero native deps. Apache 2.0 license. Your tool can read it directly or query via CLI/API.

Why Build on engram

  • Zero infrastructure. No Docker, no vector DB, no cloud. One npm install. Your users don't need to set up anything extra.
  • Zero LLM cost. Heuristic code extraction. No embeddings, no API calls, no tokens burned on indexing.
  • 10 languages. TypeScript, JavaScript, Python, Go, Rust, Java, C, C++, Ruby, PHP. Tree-sitter planned for future.
  • Survives compaction. PreCompact hook re-injects critical context. No other tool does this.
  • Self-updating graph. engram watch re-indexes on file save. Git hooks rebuild on commit. The graph is never stale.
  • Battle-tested. 520 tests. Windows + macOS + Linux. Node 20 + 22. CI on both platforms.
  • Apache 2.0. Use it commercially. Fork it. Bundle it. No restrictions.
